# Phase 5 — Hardware handoff (ESP32 implementation spec)

Everything needed to port the simulated controller to the target hardware
(ESP32 + MPU6050 + brushed DC motor + DRV8833). **No firmware is written here** —
this is the spec plus the validated gains (`results/rwip_gains.h`,
`results/rwip_gains.mat`, regenerated by `scripts/phase5_export.m`).

The exported gain is the *same* one re-validated in sim on the full realistic
chain (gyro bias+noise, accel noise, motor lag, 10-bit PWM, 200 Hz): steady
RMS θ = 0.066°, peak wheel 25.7 rad/s. So what you flash is what was proven.

---

## 1. The controller in one box

```
State    x = [theta, theta_dot, phi_dot]            (rad, rad/s, rad/s)
Control  tau_cmd = -K * x                            (N*m, wheel torque)
         K = [ -3.907814, -0.554054, -0.002694 ]     (RWIP_K_* in the header)
  i.e.   tau_cmd = 3.90781*theta + 0.55405*theta_dot + 0.002694*phi_dot
Actuate  V = (R/Kt)*tau_cmd + Ke*phi_dot             (back-EMF feed-forward)
         V = clamp(V, -V_MAX, +V_MAX)                (supply saturation)
         PWM duty = |V| / V_bus ,  direction = sign(V)   (DRV8833)
Rate     200 Hz fixed (5.000 ms), timer-driven
```

Run at 200 Hz. The gain was discretized at exactly this rate (`dlqr` on the
ZOH model) — **if you change the loop rate, re-run `phase5_export.m`** to get the
matching gain.

---

## 2. Sign & unit conventions (the reference that prevents 90% of bugs)

| Quantity | Symbol | Unit | Zero / sign |
|----------|--------|------|-------------|
| arm angle | `theta` | rad | 0 = **upright/balanced**; **+CCW** |
| arm rate | `theta_dot` | rad/s | +CCW |
| wheel speed | `phi_dot` | rad/s | wheel speed **relative to the arm**, +CCW |
| motor torque | `tau_cmd` | N·m | + spins wheel +CCW → reaction drives arm −CW |
| motor voltage | `V` | V | + drives wheel +CCW |

**Everything is SI.** Convert IMU output (often deg, deg/s) to rad, rad/s before
the controller. The single most important check on hardware: a small **+θ tilt
must produce a +V** that pushes the arm back toward upright. If the bridge spins
the wrong way the system diverges instantly — fix by flipping the motor leads or
negating `V` (and check the gyro/accel sign for θ likewise).

## 3. Signal chain

```
  MPU6050 ──I2C──> [theta fusion]  ─┐
   accel ───────> theta_acc          ├─> theta
   gyro ──(de-bias)─> theta_dot ─────┘   theta_dot
  wheel sensor ──> phi_dot ───────────────────────┐
                                                    v
                          u = -K*[theta;theta_dot;phi_dot]   (N*m)
                                                    v
                 V = R/Kt*u + Ke*phi_dot ; clamp ; PWM(sign,|V|/Vbus)
                                                    v
                                  DRV8833 ──> brushed DC motor ──> wheel
```

## 4. Sensing

### 4a. `theta` — complementary filter (accel + gyro)
The accelerometer gives an absolute tilt but is corrupted by the arm's own
acceleration during motion; the gyro is clean short-term but drifts. Fuse:

```
theta_acc  = atan2(accel_perp, accel_along) - theta_mount   // calibrate so upright = 0
theta      = ALPHA*(theta + theta_dot*dt) + (1-ALPHA)*theta_acc   // ALPHA = 0.98
```

`ALPHA = 0.98` at 200 Hz ⇒ ~0.25 s crossover: trust the gyro through transients,
let the accel correct slow drift. Pick `accel_perp/accel_along` for the two MPU
axes in the pivot plane; **calibrate `theta_mount`** by averaging `theta_acc`
with the arm physically held upright at startup.

> Caveat: `theta_acc` is only trustworthy near static. During fast balancing the
> tangential/centripetal acceleration of the wheel mount biases it — the high
> `ALPHA` is what makes this OK.

### 4b. `theta_dot` — de-biased gyro
Use the gyro axis aligned with the pivot directly: `theta_dot = gyro_rate - bias`,
sign matched to +CCW. The LQR is the `theta_dot` consumer for damping.

### 4c. `phi_dot` — wheel speed (KEY BUILD DECISION)
The state needs wheel speed relative to the arm. Options, best first:
1. **Quadrature/hall encoder on the wheel or motor shaft** (recommended): a motor
   encoder reads rotor-vs-housing speed, which *is* `phi_dot` (housing is on the
   arm). `phi_dot = (delta_counts / counts_per_rev) * 2*pi / dt`.
2. **Back-EMF estimate** (needs current sense): `phi_dot ≈ (V - i*R)/Ke`.
3. **Observer**: integrate `phi_ddot = (tau - b_w*phi_dot)/I_w - theta_ddot` from
   the commanded `tau` (drifts; use only with periodic correction).

The `phi_dot` gain is small (`-0.0027`), so a *coarse* estimate is fine for
balancing — but it is exactly the term that bleeds wheel momentum, so you need
**some** wheel-speed feedback or the wheel ratchets to saturation (Phase-3 A2).
Don't drop it.

### 4d. Startup gyro-bias calibration
Hold the rig still for ~2 s (`RWIP_GYRO_CAL_N = 400` samples), average the gyro →
`bias`, subtract forever after. In sim a 1.5°/s bias is tolerated but parks the
wheel at ~5 rad/s; de-biasing minimizes that offset.

## 5. Actuation (DRV8833)

```
V_req = R_OVER_KT * tau_cmd + KE * phi_dot     // R_OVER_KT = 33.333 V per N*m
V     = clamp(V_req, -V_MAX, +V_MAX)
duty  = min(|V| / V_bus, 1.0)                  // V_bus = measured supply (~12 V)
counts= (uint16_t) lroundf(duty * RWIP_PWM_MAX)
if (V >= 0) { ledcWrite(IN1, counts); ledcWrite(IN2, 0); }   // sign-magnitude
else        { ledcWrite(IN1, 0);      ledcWrite(IN2, counts); }
```

- **Measure `V_bus`** (battery sags under load); use the measured value for `duty`
  so the delivered voltage matches the controller's intent. Set `RWIP_V_MAX` to
  the real bus voltage.
- 10-bit PWM (ESP32 LEDC), ~20 kHz carrier (above audible, below DRV8833 limits).
- Sign-magnitude (fast decay) drive shown; slow decay also fine if re-checked.

---

## 6. Loop timing
- 200 Hz from a hardware timer ISR (or a tight `micros()` scheduler). Determinism
  matters — the gain assumes a fixed 5 ms ZOH.
- Budget per tick: I2C read MPU6050 (~0.3–0.5 ms @ 400 kHz) + fusion + `-K*x`
  (a few µs) + PWM write. Comfortable inside 5 ms. Keep I2C off the critical path
  of other blocking work.

---

## 7. Safety & startup logic
- **No swing-up.** This is a *balance* controller, valid near upright only. Enable
  control only when `|theta| < RWIP_ENABLE_WIN` (~5°); otherwise hold PWM = 0 and
  wait for the user to place the arm upright.
- **Give up if fallen:** if `|theta| > RWIP_THETA_GIVEUP` (~30°), force PWM = 0 —
  beyond the recoverable range, and it stops the motor stalling at max current.
- **Wheel soft limit:** if `|phi_dot| > RWIP_WHEEL_SOFT` (~350 rad/s) the wheel is
  near saturation/authority loss — log/limit; persistent hits mean a disturbance
  you can't reject (or a sign error pumping the wheel).
- Ramp PWM from 0 on enable; zero PWM on any fault/`NaN`.

## 8. Control-loop pseudocode

```c
#include "rwip_gains.h"

void control_isr(void) {           // fires at RWIP_LOOP_HZ
    imu_read(&ax,&ay,&az,&gx,&gy,&gz);
    float theta_dot = (gx - gyro_bias);                 // rad/s, de-biased, +CCW
    float theta_acc = atan2f(a_perp, a_along) - theta_mount;
    theta = RWIP_COMP_ALPHA*(theta + theta_dot*RWIP_LOOP_DT)
            + (1.0f-RWIP_COMP_ALPHA)*theta_acc;          // rad
    float phi_dot = wheel_speed_estimate();              // rad/s, rel. to arm

    if (fabsf(theta) > RWIP_THETA_GIVEUP || !enabled) { motor_pwm(0.0f); return; }

    float tau = -(RWIP_K_THETA*theta + RWIP_K_THETADOT*theta_dot
                  + RWIP_K_PHIDOT*phi_dot);              // N*m
    float V   = RWIP_R_OVER_KT*tau + RWIP_KE*phi_dot;    // back-EMF FF
    V = clampf(V, -RWIP_V_MAX, RWIP_V_MAX);
    motor_pwm(V);                                        // sign + |V|/Vbus -> DRV8833
}
```

`motor_pwm(0)` until `|theta| < RWIP_ENABLE_WIN` for the first time (then
`enabled = true`). `gyro_bias` and `theta_mount` are set by the 2 s startup
calibration.

## 9. If hardware behaves worse than sim
- Jittery `theta`: raise loop rate, raise `ALPHA`, or move to a small Kalman/
  complementary with accel-magnitude gating; add an explicit gyro-bias re-estimate.
- Wheel drifts up over time: improve `phi_dot` sensing, or re-confirm the gyro
  bias subtraction — a residual bias parks the wheel.
- Sluggish/oscillatory: confirm the loop truly runs at 200 Hz and `V_bus` is
  measured; if the real plant inertias differ, update `rwip_params.m` and re-run
  Phases 2→5 to regenerate `Kd`.
- The whole sim is parameterized: measure the real `m_p,l,I_arm,m_w,l_w,I_w,Kt,Ke,
  R` and drop them into `rwip_params.m`; every downstream gain/limit regenerates.
